Théorie — SOC, SIEM et architecture défensive (1h)

1.1 — Le SOC : définition et fonctionnement

Un Security Operations Center (SOC) est l'unité organisationnelle et technique chargée de surveiller en permanence la sécurité d'un système d'information. Pour SecureBank SA, c'est la dernière ligne de défense avant qu'une intrusion ne devienne une brèche de données bancaires.

Le SOC répond à une réalité simple : aucun système n'est inviolable. La question n'est pas "si" une attaque se produira, mais "quand" et "combien de temps restera-t-elle invisible". Le temps moyen de détection d'une intrusion (MTTD) est de 197 jours dans les organisations sans SOC. Avec un SOC mature, il descend à quelques heures.

🎯 Analogie bancaire directe
SecureBank SA a des caméras dans ses agences physiques, des gardes, des coffres avec alarmes. Le SOC, c'est l'équivalent pour l'infrastructure numérique : Suricata = les caméras aux entrées du réseau, Wazuh = les capteurs de mouvement à l'intérieur des serveurs, Elastic SIEM = le centre de vidéosurveillance qui centralise tout, et l'analyste Tier 1 = le vigile qui regarde les écrans 24h/24. Sans ce dispositif, l'attaquant qui a pénétré le réseau peut rester invisible pendant des mois.

Les 3 niveaux d'analystes SOC

1.2 — Le SIEM : intelligence centrale du SOC

Le SIEM (Security Information and Event Management) est le système qui transforme des millions de lignes de logs bruts en alertes exploitables. Sans SIEM, un analyste recevrait chaque jour des centaines de milliers d'événements de dizaines de sources différentes — impossible à traiter manuellement.

Pour SecureBank SA, le SIEM reçoit simultanément : les logs de pfSense FW-DMZ (règles firewall franchies), les alertes Suricata (trafic réseau suspect), les logs Wazuh de chaque serveur (tentatives SSH, modifications de fichiers), et les logs Apache/nginx (requêtes web anormales). Sans corrélation, l'attaque de ce matin serait restée invisible.

🏦 Analogie bancaire
Dans une agence physique, vous avez des dizaines de caméras, des capteurs sur les coffres, des lecteurs de badges, des capteurs de mouvement. Seul, un vigile ne peut pas regarder 40 écrans en même temps. Le SIEM, c'est le logiciel qui fusionne tous ces flux et vous dit : "Caméra 3 + capteur coffre A + badge invalide = alarme". Un seul événement = bénin. Les trois combinés en 2 minutes = alerte prioritaire. C'est la corrélation.

📥 Collecte — Tous les logs vers un point central

Le SIEM ingère les logs de toutes les sources de l'infrastructure SecureBank SA. Ces logs arrivent dans des formats radicalement différents :

Trois formats différents, une seule IP source — le SIEM va les relier automatiquement.

🔄 Normalisation ECS — Un langage commun

L'ECS (Elastic Common Schema) traduit tous ces formats en champs standardisés. Résultat : une seule requête KQL permet d'interroger simultanément Suricata, Wazuh et Apache.

Après normalisation ECSKibana Discover
# Tous ces logs partagent maintenant les MÊMES champs :

source.ip           : "185.220.101.47"   # IP attaquant — uniforme
destination.ip      : "192.168.50.10"    # Serveur web SecureBank
event.module        : "suricata"         # ou "wazuh", "apache"
event.outcome       : "failure"          # uniforme sur toutes les sources
network.transport   : "tcp"

# Requête KQL unifiée dans Kibana :
source.ip: "185.220.101.47" AND event.outcome: "failure"
# → affiche TOUS les événements liés à cet attaquant, toutes sources confondues
💡

Sans ECS : l'enfer des formats

Sans normalisation, "l'IP source" s'appellerait src_ip dans Suricata, client_ip dans Apache, et remote_addr dans nginx. Impossible d'écrire une règle de corrélation cohérente. ECS résout ce problème en imposant source.ip pour tout le monde.

IDS vs IPS — Différence critique pour SecureBank

Cette distinction est fondamentale dans un contexte bancaire. SecureBank SA ne peut pas se permettre de bloquer du trafic légitime — les virements de clients, les connexions des sysadmins, les flux inter-agences. C'est pourquoi le choix IDS/IPS doit être réfléchi.

IDS passif vs IPS inline — Impact sur le trafic SecureBank
ℹ️

IDS = observer, pas bloquer

En mode IDS, Suricata voit une copie du trafic (miroir). Il alerte sans interférer. Le paquet SQLi atteint quand même l'API. Mais l'analyste reçoit l'alerte en temps réel. Avantage : aucun risque de couper un virement légitime. C'est le mode utilisé en Séance 1 — on apprend à voir avant d'agir.

🚨

IPS — Puissant mais risqué sans tuning

En IPS, Suricata est en coupure : il analyse chaque paquet AVANT qu'il arrive à destination. Si une règle est trop sensible, un virement légitime de 50 000 DH peut être bloqué. Dans un contexte bancaire, cela est inacceptable. D'où la règle d'or : IDS d'abord → tuner les règles → IPS ensuite. Ce mode sera activé en Séance 3 après 2 séances d'observation.

📦 Filebeat — Le collecteur temps réel

Filebeat est l'agent léger déployé sur chaque serveur (web01, api01, proxy). Il surveille les fichiers de log en temps réel (tail -f intelligent) et envoie chaque nouvelle ligne vers Elasticsearch avec latence < 1 seconde.

yaml/etc/filebeat/filebeat.yml — SecureBank
filebeat.modules:
  - module: suricata     # alertes IDS pfSense
  - module: apache       # logs portail web
  - module: nginx        # logs reverse proxy
  - module: system       # auth.log, syslog

filebeat.inputs:
  - type: syslog          # reçoit syslog pfSense
    protocol.udp.host: "0.0.0.0:514"
    tags: ["pfsense", "securebank-fw"]

output.elasticsearch:
  hosts: ["192.168.10.100:9200"]  # SIEM LAN
  index: "securebank-%{[agent.name]}-%{+yyyy.MM.dd}"

setup.kibana.host: "192.168.10.100:5601"
setup.dashboards.enabled: true  # dashboards Suricata prêts

📊 Kibana SIEM — L'écran de l'analyste

Kibana Security est l'interface que l'analyste SOC de SecureBank regarde en permanence. Les 5 vues essentielles :

  • Security → Overview : tableau de bord global — tendances d'alertes, top attaquants, événements récents
  • Security → Alerts : liste priorisée des alertes actives avec severity, MITRE ATT&CK, source IP
  • Security → Rules : gérer les règles EQL/KQL de détection. C'est ici qu'on crée la règle brute-force
  • Security → Timeline : reconstituer la chronologie de l'attaque pour le rapport IR
  • Discover : requêtes libres KQL sur l'ensemble des logs — pour l'investigation ad hoc
KQLKibana Discover — Requêtes essentielles
# Tous les événements de l'attaquant
source.ip: "185.220.101.47"

# Alertes Suricata critique/haute depuis 1h
event.module: suricata AND suricata.eve.alert.severity: (1 OR 2)

# Brute-force SSH (Wazuh)
event.module: wazuh AND rule.groups: authentication_failed

# Requêtes POST suspectes sur l'API paiement
url.path: "/api/login" AND http.request.method: POST AND http.response.status_code: 400

🤖 Wazuh — La surveillance hôte

Wazuh surveille l'intérieur de chaque serveur du DMZ SecureBank. Trois fonctions critiques dans ce scénario :

  • Log Analysis : lit /var/log/auth.log et déclenche la règle 5712 "SSH brute-force attack" après 8 échecs depuis la même IP
  • FIM (File Integrity Monitoring) : si l'attaquant réussit à se connecter et modifie /etc/passwd ou un fichier PHP, Wazuh génère une alerte critique immédiate
  • Active Response : configuré en séance 3 — Wazuh peut exécuter automatiquement iptables -A INPUT -s 185.220.101.47 -j DROP après détection de brute-force

sequence by source.ip with maxspan=30m

  /* Phase 1 : brute-force SSH */
  [authentication where event.outcome == "failure"
   and destination.port == 22]

  [authentication where event.outcome == "failure"
   and destination.port == 22]

  [authentication where event.outcome == "failure"
   and destination.port == 22]

  /* Phase 2 : tentative SQLi sur l'API */
  [network where event.module == "suricata"
   and suricata.eve.alert.category like "*sql*"]
